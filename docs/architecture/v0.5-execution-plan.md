## Goal

Ship Animus v0.5: a kernel + flavors architecture that defines four new plugin kinds at the protocol level (`workflow_runner`, `queue`, `durable_store`, `memory_store`), implements one reference plugin per kind, and updates the daemon to call those plugins instead of in-tree code. v0.5 ships demo-quality, not production-hardened.

**Definition of "v0.5 done":**

1. `animus-protocol` workspace tagged `v0.5.0` with four new protocol crates + extended `animus-plugin-protocol` + re-homed subject types
2. Four reference plugin repos exist under `launchapp-dev` and tagged `v0.1.0-dev`:
   - `animus-workflow-runner-default` (Rust lift)
   - `animus-queue-default` (Rust lift)
   - `animus-step-durable-dbos` (TypeScript, DBOS Option A)
   - `animus-memory-zep` (TypeScript, Zep Cloud)
3. `ao-cli` `v0.5` branch routes workflow execution and queue operations through plugin host calls instead of in-tree code
4. `flavors/default.toml` exists; `animus flavor` CLI subcommand works
5. `animus daemon health` surfaces the active flavor identifier
6. End-to-end demo: kill the daemon mid-workflow; restart; durable_store recovers; agents resume
7. Every artifact has passed codex self-vet with zero `[P1]` findings before commit
8. SpecStory is NOT shipped (no public ingestion API; deferred per `kernel-and-flavors.md`)

## Discipline rules (binding across all waves)

From `kernel-and-flavors.md` §"Discipline rules":

1. **One flavor at launch.** Only `default`. No second flavor scaffolding even if it's "easy."
2. **Kernel stays sacred.** No new kernel responsibilities outside the four new plugin kinds. Anything else gets pushed to plugins, packs, or workflow YAML.
3. **Flavors are opinionated, not configurable.** No install-time wizards.
4. **Marketing layer ≠ architecture layer.** Do not edit `ao-cli`'s README or `launchapp.dev` landing copy.
5. **No new sibling products.** No git service, no task management product, no LaunchApp adjacencies.
6. **Push down a tier.** When a capability is proposed, the default answer is "can it be a workflow YAML phase, a command, a skill, or a community script?" If yes — it is NOT a plugin kind. New plugin kinds require multi-RPC stateful protocols, lifecycle, or event-driven long-running behavior.

**Sub-discipline for this release specifically:**

7. **Demo-quality, not production-hardened.** The four hard problems from `animus-vs-dbos-transact.md` (idempotency across IPC, agent reattach, deterministic LLM-decision replay, schema sync) are explicit deferrals. Solve the happy path with documented limits.
8. **No `workspace = true` for cross-repo dependencies.** Every plugin repo is standalone Cargo. Use git-tagged dependencies (`git = "..." tag = "v0.5.0"`) for protocol crates and `ao-cli` shared types.
9. **All method names snake_case.** No hyphens in `METHOD_*` constants.
10. **Subject types live in `animus-subject-protocol` only.** `ao-cli`'s `protocol` crate re-exports for back-compat; nothing else defines them.

## Orchestration model — you are the swarm coordinator

You are not a single-wave executor. **You drive Wave 1 through Wave 4 to completion.** You decide what to do yourself, what to dispatch to sub-agents, and when to wait.

**Self-execute** (single agent, sequential):
- Wave 1 (one workspace, one branch — no parallelism gain from subagents)
- Wave 3 (one repo, one branch)

**Dispatch sub-agents** (parallel work, independent repos):
- Wave 2A: dispatch one Agent for Brief E and one for Brief F simultaneously
- Wave 2B: dispatch one Agent for Brief B and one for Brief C simultaneously
- Wave 2A and 2B can run in the same batch — fire all four briefs in parallel after Wave 1 tag

Use the `Agent` tool with `subagent_type: "general-purpose"` for each plugin-implementation brief. Each subagent gets a self-contained prompt that:

1. Points at this execution plan + the spec doc as canonical references
2. Specifies the brief number (E, F, B, or C) and its scope
3. Instructs the subagent to follow the same discipline rules + codex self-vet procedure
4. Defines its acceptance gate (the wave's gate from this plan)
5. Demands a final report in the standard shape (see "Final report" section)

**You wait synchronously** for the parallel batch to complete (the Agent tool returns when each subagent finishes). When all four return, you read their reports, verify their tags exist on origin, append the tag URLs to the "Wave handoff state" section at the bottom of this plan, then proceed to Wave 3.

**State threading:** every wave's output becomes the input for the next wave. The Wave 1 tag URL → cited in Briefs B/C/E/F's Cargo deps. The Wave 2A tag URLs → cited in Wave 3's `flavors/default.toml`. You maintain that state by **appending to the "Wave handoff state" section at the bottom of this file** after each wave completes. Wave 3 and Wave 4 sub-agents read it to know what's installable.

**If a sub-agent fails:** read its final report. If the failure is a P1 codex finding it couldn't resolve, re-dispatch with a refined prompt that calls out the specific finding. If the failure is a scope drift (the agent added work outside the brief), re-dispatch with stricter scope language. Do not silently accept partial work.

### Wave 2B — TypeScript greenfield (Briefs B + C, in parallel after Wave 1 tag)

**Repos:** new `launchapp-dev/animus-step-durable-dbos` + `launchapp-dev/animus-memory-zep`
**Branch:** `v0.1.0-dev` in each
**Language:** TypeScript, Node ≥20

**Brief B — `animus-step-durable-dbos`:**

DBOS Option A from the design analysis. Implements `durable_store` plugin kind.

- Use `@dbos-inc/dbos-sdk ^4.18`. Postgres via Docker Compose snippet shipped in repo `examples/`.
- Implement reservation table (`animus_step_reservations`) with `released BOOLEAN` column and partial unique index `WHERE released = FALSE`. (Postgres won't allow `now()` in index predicates.)
- Implement commits table (`animus_step_commits`) with explicit `outcome TEXT CHECK (outcome IN ('success', 'error'))` column.
- `begin_step` single transaction: sweep expired → check `animus_step_commits` FIRST (returns ALREADY_COMMITTED or PRIOR_ERROR) → check live reservations (returns IN_PROGRESS) → INSERT new reservation (returns NEW).
- `commit_step` single transaction: DBOS step write + INSERT INTO commits (ON CONFLICT DO NOTHING) + UPDATE reservations SET released = TRUE.
- `abandon_step` UPDATEs `released = TRUE` (only affects active reservations; does NOT clear committed errors — PRIOR_ERROR is terminal).
- Background tokio task sweeping expired reservations every 30s.
- Integration tests: replay returns ALREADY_COMMITTED; kill mid-execution recovers via `recover_in_flight`.
- Demo script: kill -9 daemon mid-workflow → restart → durable_store recovers → agents resume.

**Brief C — `animus-memory-zep`:**

Implements `memory_store` plugin kind via Zep Cloud.

- Use `@getzep/zep-cloud ^3.23`. Env: `ZEP_API_KEY` (required), `ZEP_BASE_URL` (optional BYOC).
- Scope → graphId mapping via `normalize()` function (lowercase, [a-z0-9_-] only, collapse `-`, 64-char limit, sha256 fallback). Defined exactly in spec §4.
- `put`: `graph.create()` (idempotent on 409), then `graph.add({ data: JSON.stringify({key, value}), metadata: { key, ttl_s } })`. Return `indexed_immediately: false` (Zep is async) + Zep episode UUID as `record_id`.
- `get`: `graph.search({ limit: 1 })` with metadata key filter. Document the search-based fallback.
- `query`: `graph.search({ scope: "edges", limit: top_k clamped to 50, reranker: "rrf" })`.
- `list_scopes`: cursor-paginated `graph.listAll`. Filter by normalized prefix (`proj_${normalize(project_id)}`), NEVER raw `project_id`.
- `delete_scope`: `graph.delete(graphId)`.
- Capabilities: `{ native_ttl: false, native_key_get: false, strong_consistency: false, max_query_top_k: 50 }`.
- Integration test against real Zep Cloud dev account.

**Acceptance gate (both briefs):**

## Failure modes to avoid

These are the patterns that have killed previous Animus releases. Recognize and abort:

1. **Scope creep on the next thing.** "I'm building queue, but workflow_runner needs Y first" -> no, ship queue as specified, defer Y to v0.6.
2. **Trying to solve the four hard problems.** Idempotency across IPC, agent reattach, deterministic LLM-decision replay, schema sync. These are deferrals. Document; do not solve in v0.5.
3. **Adding "while I'm here" cleanups.** That CLI subcommand that looks ugly → leave it alone. Wave 3 has specific deletions; nothing else.
4. **Inventing protocol fields.** If the spec doc doesn't name a field, do not add one. Open a follow-up PR for v0.6.
5. **Using `workspace = true` for cross-repo deps.** Will not resolve in standalone plugin repos.
6. **Method names with hyphens.** All `snake_case`. The lint check at the top of acceptance criteria catches this.
7. **Skipping codex self-vet to ship faster.** Six rounds of codex caught 18 P1s in the spec. Codex on the implementation will catch more. Always run it. Always fix all P1s.
8. **Editing the architecture docs to match buggy code.** Code conforms to spec; not the other way around. If a spec field seems wrong, raise it in the final report and STOP — do not modify the spec without explicit human sign-off.
9. **Touching ao-cli README marketing.** Human-owned. Do not edit.
10. **Trying to ship SpecStory.** Explicitly deferred. SpecStory has no public ingestion API.
