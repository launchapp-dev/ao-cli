# Migration: v0.4 → v0.5

> **Status (2026-05-28):** P3 audit follow-ups, including the
> `ao-workflow-runner` → `animus-workflow-runner` rename, are landing in
> trailing v0.4.x patches ahead of the v0.5 line. This document collects
> the user-visible naming changes so operators can plan their upgrade.

## Workflow runner binary rename

### What changed

The standalone phase runner produced by the `workflow-runner-v2` crate has
been renamed at the binary layer:

| Era | Binary name | Where it lives |
|---|---|---|
| v0.4.0 → v0.4.15 (`ao` era follow-up) | `ao-workflow-runner` | `~/.local/bin/` (installer); `/usr/local/bin/` (Docker image) |
| v0.4.16 → v0.5 | `animus-workflow-runner` | same locations |

This closes the v0.4.0 known-follow-up that left the legacy `ao-*` bin
name in place after the `ao` → `animus` rename. The Cargo crate directory
name (`crates/workflow-runner-v2`) is unchanged — only the `[[bin]] name`
in `Cargo.toml` and the binary file name on disk move.

### Back-compat window

The rename ships with two compatibility mechanisms so that mixed-version
installations keep working through the upgrade:

1. **Installer-created symlink.** `scripts/install.sh` creates an
   `ao-workflow-runner` symlink that points at `animus-workflow-runner`
   in the same install directory. Operators running a v0.4.x daemon PID
   that still spawns `ao-workflow-runner` keep dispatching workflows
   after they install the new release tarball, because the legacy name
   still resolves via the symlink.

2. **Daemon-side fallback resolver.** The v0.4.16+ daemon's runner
   resolver in
   `crates/orchestrator-daemon-runtime/src/dispatch/build_runner_command_from_dispatch.rs`
   probes `animus-workflow-runner` first and falls back to
   `ao-workflow-runner` if the new name is not present. Resolution order
   for each name: sibling of `current_exe` first, then `PATH` search via
   `which`-style scan. The PATH fallback handles the "daemon binary in
   a different prefix than the runner" case (e.g. `cargo install animus`
   put the daemon in `~/.cargo/bin/` while a prior v0.4.x installer left
   `ao-workflow-runner` in `~/.local/bin/`). When the resolver lands on
   the legacy name it emits a `tracing::warn!` line so the upgrade nudge
   is loud:

   > `found legacy ao-workflow-runner; reinstall with 'curl -fsSL https://raw.githubusercontent.com/launchapp-dev/animus-cli/main/scripts/install.sh | sh' to upgrade`

   This handles the inverse case: the daemon binary has been upgraded
   in place (`cargo install`, manual copy) but the user has not rerun
   the installer to land the new runner binary. The daemon keeps
   working off the legacy binary until the installer is rerun.

The back-compat window will be preserved through the entire v0.4.x line.
v0.5 is the earliest release that may remove the legacy fallback; the
removal will be called out in `v0.5-to-v0.6.md` ahead of time.

### Recommended upgrade

1. Stop the running daemon.

   ```bash
   animus daemon stop
   ```

2. Reinstall the CLI tarball. The installer will overwrite the
   existing binaries in `~/.local/bin/` and create the
   `ao-workflow-runner → animus-workflow-runner` symlink.

   ```bash
   curl -fsSL https://raw.githubusercontent.com/launchapp-dev/animus-cli/main/scripts/install.sh | sh
   ```

3. Confirm both names resolve to the same binary:

   ```bash
   ls -l ~/.local/bin/ao-workflow-runner   # symlink → animus-workflow-runner
   ~/.local/bin/animus-workflow-runner --help | head -1
   ```

   (`animus-workflow-runner` is a clap subcommand binary — use `--help`
   instead of `--version`; macOS `readlink` does not support `-f`, so
   `ls -l` is the portable check for the symlink target.)

4. Restart the daemon:

   ```bash
   animus daemon start --auto-install
   ```

5. Tail the daemon log briefly to confirm there are no
   `found legacy ao-workflow-runner` warnings — that's the signal that
   either the symlink is missing or the install directory still has a
   stale standalone `ao-workflow-runner` file.

### Custom installs

If you install Animus by copying binaries out of the release tarball
manually (e.g. into `/opt/animus/bin/`), you must also create the
back-compat symlink so that v0.4.x daemon PIDs keep dispatching while
they coexist with the new binary:

```bash
cd /opt/animus/bin
ln -s animus-workflow-runner ao-workflow-runner
```

The daemon-side fallback resolver only ships in v0.4.16+. A v0.4.x
daemon PID that's still running has no fallback and will hard-fail to
spawn `ao-workflow-runner` if you remove the legacy name from disk. The
symlink keeps those PIDs alive until they're restarted on the new
daemon binary; only then is the resolver fallback in effect and only
then is it safe to drop the symlink.

### Container deployments

The shipped `Dockerfile` creates the symlink inside the image at
`/usr/local/bin/ao-workflow-runner → animus-workflow-runner`, so
`ghcr.io/launchapp-dev/animus:latest` is back-compat without any
operator action. If you build your own image from the workspace, follow
the same pattern (copy `animus-workflow-runner`, then `ln -s`).

### Why both layers?

The rename touches two independent surfaces — the binary on disk and the
PATH lookup inside the daemon. Either surface alone is not enough:

- **Symlink only**: an upgraded daemon would always look up
  `ao-workflow-runner`, never `animus-workflow-runner`. The symlink
  works but the daemon never adopts the new name, so the rename never
  finishes.
- **Resolver fallback only**: an upgraded installer drops only the new
  binary. A v0.4.x daemon PID that's still running keeps spawning
  `ao-workflow-runner`, which is now gone — workflows fail to start.

Together they cover both directions of the in-progress upgrade.
